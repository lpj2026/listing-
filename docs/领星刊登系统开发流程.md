## 1. 项目背景与目标

### 1.1 背景

当前运营人员使用 **店小秘**（`https://www.dianxiaomi.com/web/amazon/add`）创建亚马逊商品，填写刊登资料后发布到亚马逊 Seller Central「管理所有库存」。公司内部同时使用 **领星 ERP** 管理 Listing、库存、订单和本地 SKU 配对。

两套系统数据最终都指向亚马逊，但刊登入口分散在店小秘，不利于：

- 统一业务流程与字段规范
- 与内部系统（采购、库存、品名）深度集成
- 降低第三方 ERP 订阅成本
- 按业务需求定制表单、校验和自动化

### 1.2 项目目标

| 目标 | 说明 |
|------|------|
| 替代店小秘刊登 | 自研刊登表单，覆盖新品刊登与已有商品更新 |
| 对接领星 API | 通过领星开放 API 完成分类查询、Schema 获取、资料提交、结果轮询 |
| 保持数据同步 | 刊登成功后，商品自动出现在亚马逊后台，并由领星 Listing 同步 |
| 自动 SKU 配对 | 刊登成功后，自动或半自动完成 MSKU ↔ 领星本地 SKU 配对 |
| 部署在阿里云 | 服务端部署于 `8.137.177.25`，满足领星 IP 白名单要求 |

### 1.3 不在本期范围

- 替换领星 ERP 本身（库存、订单、财务仍用领星）
- 多平台刊登（本期仅亚马逊）
- 广告投放、跟卖监控等运营功能

## 2. 现状与痛点

### 2.1 当前流程（店小秘）

```
运营人员
  → 店小秘刊登表单（手工填写）
  → 店小秘调用亚马逊 SP-API
  → 亚马逊 Seller Central「管理所有库存」
  → 领星 Listing 从亚马逊 All Listing 报表同步（15分钟~1小时）
  → 运营在领星手动做 MSKU ↔ 本地 SKU 配对
```

### 2.2 目标流程（自研系统）

```
运营人员
  → 自研刊登系统（表单 + 校验 + 草稿）
  → 领星刊登开放 API（5 步流程）
  → 亚马逊 Seller Central
  → 领星 Listing 自动同步
  → 自研系统轮询刊登结果 + 自动 SKU 配对
```

### 2.3 关键认知纠正

| 误解 | 正确理解 |
|------|----------|
| 「上传到领星 Listing」 | Listing 是亚马逊数据的**只读镜像**，不能直接写入 |
| 「领星没有刊登 API」 | 领星提供完整的 **5 步刊登开放 API**（`/basicOpen/` 和 `/listing/publish/`） |
| 「必须直连亚马逊 SP-API」 | 可通过领星 API 间接调用，领星已封装 SP-API 复杂度 |
| 「同步会影响刊登」 | 同步是从亚马逊 → 领星拉取，是刊登成功后的正常环节，无负面影响 |

## 3. 核心结论（可行性）

### 3.1 可行性评估：✅ 完全可行

已于 **2026-06-05** 在阿里云服务器 `8.137.177.25` 完成在线 API 验证：

| 测试项 | 结果 |
|--------|------|
| Token 获取 | ✅ 成功 |
| 查询店铺（133 个） | ✅ 成功 |
| 查询 Listing（只读） | ✅ 成功 |
| 查询本地产品 | ✅ 成功 |
| 刊登-查询根分类 | ✅ 成功（参数 `storeId`） |
| 刊登-查询刊登结果 | ✅ 成功 |
| 刊登-发布商品 | ⏳ 未实测（需完整商品资料，建议在开发阶段用测试 SKU 验证） |

### 3.2 技术路线确认

**推荐方案：自研刊登 UI + 领星刊登开放 API**

不推荐初期直连亚马逊 SP-API，原因：

- 领星已封装 Schema 获取、分类树、刊登队列、结果查询
- 与领星 Listing 同步、SKU 配对天然衔接
- 减少 SP-API 授权、Product Type Definitions 独立维护成本

## 5. 数据流与业务链路

### 5.1 新品刊登完整链路

```
[1] 选择店铺 + 站点
      ↓
[2] 浏览亚马逊分类树 → 确定末级分类 → 获取 productType
      ↓
[3] 获取该 productType 的 JSON Schema（propertiesZh 中文版）
      ↓
[4] 运营填写商品资料（标题、品牌、图片、属性、价格等）
      ↓
[5] 本地校验 → 保存草稿（可选）
      ↓
[6] 调用 publish 接口提交（operationType=0）
      ↓
[7] 获取 record_unique_id → 轮询刊登结果
      ↓
[8] status=1 刊登成功
      ↓
[9] 等待领星 Listing 同步（15分钟~1小时）
      ↓
[10] 调用领星 API 查询 Listing 确认 MSKU 出现
      ↓
[11] 调用领星 API 完成 MSKU ↔ 本地 SKU 配对
```

### 5.2 更新已有商品链路

```
[1] 输入 MSKU → 调用 product/search 获取已有商品信息和 productType
      ↓
[2] 仅需修改的字段填入 attributes（未填字段保留原值）
      ↓
[3] 若需填写新字段 → 获取完整 JSON Schema
      ↓
[4] 调用 publish 接口（operationType=1）
      ↓
[5] 轮询刊登结果
```

### 5.3 与领星 Listing 同步的关系

| 时机 | 亚马逊 | 领星 Listing |
|------|--------|-------------|
| 提交刊登后 | 处理中（约 10 分钟） | 尚未出现 |
| 刊登成功（status=1） | 已有 MSKU | 可能尚未同步 |
| 同步完成后 | 已有 | 可搜索到 MSKU、ASIN、FNSKU |

**注意**：刊登成功 ≠ Listing 立即可见。需额外等待领星从亚马逊 All Listing 报表同步。

## 6. 环境准备与部署

1. 登录领星 ERP 后台

4. 获取 `AppID` 和 `AppSecret`

## 7. API 鉴权机制

### 7.1 获取 Token

```
POST /api/auth-server/oauth/access-token?appId={appId}&appSecret={urlEncodedSecret}
```

响应：

```json
{
  "code": "200",
  "msg": "OK",
  "data": {
    "access_token": "e6415f7e-1554-4f37-a00c-fa2f5478ea25",
    "refresh_token": "535ce050-25e5-4782-9d99-f49c70216566",
    "expires_in": 541
  }
}
```

### 7.2 业务接口签名

每次请求需携带 Query 参数：

| 参数 | 说明 |
|------|------|
| `app_key` | AppID |
| `access_token` | 上一步获取的 Token |
| `timestamp` | 当前 Unix 时间戳（秒） |
| `sign` | 签名值 |

**签名算法：**

1. 合并所有 Query 参数 + POST Body 参数为一个 dict
2. 按 key 字母序排序
3. 拼接为 `key1=value1&key2=value2&...`（value 为复杂类型时 JSON 序列化）
4. 对拼接字符串做 MD5，转大写
5. 用 AES-ECB 加密（key = AppID），再 Base64 编码 → 得到 `sign`

### 7.3 Token 管理策略

- Token 有效期约 9 分钟（`expires_in` 540 秒左右）
- 建议在 Token 剩余 1/5 时间时主动刷新
- 使用 Redis 缓存 Token，避免每次请求都重新获取
- 收到 `2001003`（token 过期）错误时自动重新获取并重试

### 7.4 API 基础地址

| 类型 | 基础地址 | 路径前缀 |
|------|----------|----------|
| 鉴权 | `https://openapi.lingxing.com` | `/api/auth-server/` |
| 刊登 API | `https://openapi.lingxing.com` | `/basicOpen/` 或 `/listing/publish/` |
| 基础数据 API | `https://openapi.lingxing.com` | `/erp/sc/` |

**重要**：刊登 API **不带** `/erp/sc/` 前缀。

## 8. 刊登核心流程（5 步）

### 步骤一：确认商品 productType

#### 8.1.1 新品刊登 — 查询分类

**第一步：查询根分类**

```
POST /basicOpen/openapi/publish/manage/categoryRoot
```

请求参数（已实测）：

```json
{
  "storeId": 12518
}
```

> ⚠️ 参数名是 `storeId`（驼峰），不是 `sid` 或 `store_id`。

响应关键字段：

| 字段 | 说明 |
|------|------|
| `categoryUniqueId` | 分类唯一 ID，用于查子分类 |
| `categoryName` | 分类名称 |
| `hasChildren` | 是否有子分类（1=有，0=无） |
| `productTypeOrigin` | 商品类型（可能在中间层级出现） |

**第二步：递归查询子分类**

```
POST /basicOpen/openapi/publish/manage/categoryChildren
```

请求参数：

```json
{
  "storeId": 12518,
  "categoryUniqueId": "107883964420587523"
}
```

流程：

```
根分类 → 子分类 → 子分类 → ... → 末级分类（hasChildren=0）
```

末级分类需留存：

| 字段 | 用途 |
|------|------|
| `productTypeOrigin` | 下一步获取 Schema 的 productType |
| `browseNodeAttributes` | 第四步构建 attributes 时使用 |

站点差异：

| 站点 | 填入字段 |
|------|----------|
| 美国站 | `item_type_keyword`（取自 browseNodeAttributes；无则用 productType 兜底） |
| 非美国站 | `recommended_browse_nodes` |

#### 8.1.2 更新已有商品 — 查询商品信息

```
POST /listing/publish/openapi/amazon/product/search
```

请求参数：

```json
{
  "sellerId": "AUF98RFGA31T9",
  "marketplaceId": "ATVPDKIKX0DER",
  "skus": ["MLYR-US-230722-0828"]
}
```

> ⚠️ 参数名是 `skus`，不是 `mskus`（实测 `mskus` 报参数错误）。

响应关键字段：

| 字段路径 | 说明 |
|----------|------|
| `info.productTypes.productType` | 该 MSKU 的 productType |
| `info.attributes` | 目前已填写的全部属性，更新时可复用结构 |

### 步骤二：获取 JSON Schema

```
POST /basicOpen/openapi/publish/manage/getProductType
```

请求参数：

```json
{
  "marketplaceId": "ATVPDKIKX0DER",
  "productTypeOrigin": "AUTO_PART"
}
```

响应：

| 字段 | 说明 |
|------|------|
| `properties` | 英文版表单定义 |
| `propertiesZh` | **中文版**表单定义（前端优先使用） |

Schema 结构：

```json
{
  "properties": {
    "item_name": { "type": "array", "items": { ... }, "title": "商品名称" },
    "brand": { ... },
    "bullet_point": { ... }
  },
  "allOf": [
    { "if": { ... }, "then": { "required": ["color"] } }
  ]
}
```

使用场景：

| 场景 | 是否需要 Schema |
|------|----------------|
| 新品刊登 | ✅ 必须获取完整 Schema |
| 更新已有商品（仅改已填字段） | ❌ 可跳过，用 search 返回的 attributes 结构 |
| 更新已有商品（需填新字段） | ✅ 需获取完整 Schema |

#### 8.2.1 FBM 运费模板（新品 FBM 配送必填）

```
POST /basicOpen/openapi/publish/manage/getMerchantShippingGroup
```

请求参数：

```json
{
  "seller_id": "AUF98RFGA31T9",
  "marketplace_id": "ATVPDKIKX0DER",
  "product_type": "AUTO_PART",
  "flag": 0
}
```

| flag | 说明 |
|------|------|
| `0` | 返回已缓存的运费模板（快，优先使用） |
| `1` | 实时从亚马逊后台拉取（慢） |

响应：`name`（模板名称）、`value`（模板 ID，填入 `merchant_shipping_group` 字段）

### 步骤三：本地处理 Schema，填写商品资料

#### 8.3.1 表单组成

每份刊登资料包含：

| 组成部分 | 说明 |
|----------|------|
| `properties` | 所有字段的名称、类型、限制、枚举值 |
| `allOf` | 条件判断（某些条件下字段变为必填） |
| `attributes` | 最终提交给亚马逊的 JSON 数据 |

#### 8.3.2 attributes 数据结构规范

每个属性值为**数组**，数组元素为对象：

```json
{
  "item_name": [
    {
      "value": "Soklopi R0445200 Backplate Replacement Kit",
      "language_tag": "en_US",
      "marketplace_id": "ATVPDKIKX0DER"
    }
  ],
  "brand": [
    {
      "value": "Soklopi",
      "language_tag": "en_US",
      "marketplace_id": "ATVPDKIKX0DER"
    }
  ],
  "purchasable_offer": [
    {
      "our_price": [
        {
          "schedule": [
            {
              "value_with_tax": 110.99
            }
          ]
        }
      ],
      "marketplace_id": "ATVPDKIKX0DER"
    }
  ]
}
```

#### 8.3.3 前端动态表单渲染逻辑

```
1. 解析 propertiesZh → 生成基础表单字段
2. 解析 allOf 条件 → 动态显示/隐藏/必填字段
3. 枚举字段 → 渲染为下拉选择
4. 数组字段 → 支持添加/删除多行
5. 图片字段 → 对接 OSS 上传
6. 实时校验 → 基于 Schema 的 type、minLength、maxLength、enum 等
```

推荐工具：

- JSON Schema 表单生成：`@rjsf/core`（React JSON Schema Form）
- 或基于 Vue 自行解析 Schema 渲染

#### 8.3.4 更新模式注意事项

- **仅填写需要修改的字段**
- 未填写的字段将保留亚马逊上的原值
- 不要误传空值，否则可能覆盖已有数据

### 步骤四：提交商品资料

```
POST /listing/publish/openapi/amazon/product/publish
```

请求体：

```json
{
  "store_id": 12518,
  "data": [
    {
      "sku": "MLYR-US-NEW-001",
      "productType": "AUTO_PART",
      "operationType": 0,
      "attributes": {
        "item_name": [
          {
            "value": "Product Title Here",
            "language_tag": "en_US",
            "marketplace_id": "ATVPDKIKX0DER"
          }
        ],
        "brand": [
          {
            "value": "Soklopi",
            "language_tag": "en_US",
            "marketplace_id": "ATVPDKIKX0DER"
          }
        ],
        "supplier_declared_has_product_identifier_exemption": [
          {
            "value": true,
            "marketplace_id": "ATVPDKIKX0DER"
          }
        ],
        "item_type_keyword": [
          {
            "value": "automotive-replacement-parts",
            "marketplace_id": "ATVPDKIKX0DER"
          }
        ]
      }
    }
  ]
}
```

参数说明：

| 字段 | 说明 |
|------|------|
| `store_id` | 店铺 ID（下划线命名） |
| `sku` | MSKU（卖家 SKU） |
| `productType` | 步骤一获取的 productType |
| `operationType` | `0`=新品刊登，`1`=更新已有商品 |
| `attributes` | 按 Schema 构建的商品属性 JSON |

能力：

- ✅ 支持批量提交（同店铺多个 SKU）
- ✅ 支持混合提交（新品 + 更新在同一请求中）
- ⚠️ 不同 productType 的商品需分别构建 attributes 后合并提交

响应：

```json
{
  "code": 1,
  "msg": "成功",
  "data": {
    "record_unique_id": "xxxxxxxx"
  }
}
```

> 务必保存 `record_unique_id`，用于步骤五轮询。

### 步骤五：查询刊登结果

```
POST /listing/publish/openapi/amazon/product/list
```

请求参数（多种查询方式，多参数取交集）：

```json
{
  "record_unique_id": "xxxxxxxx",
  "store_id": 12518,
  "sku": "MLYR-US-NEW-001",
  "offset": 0,
  "length": 20
}
```

响应 status 含义：

| status | 含义 | 处理方式 |
|--------|------|----------|
| `0` | 刊登处理中 | 继续轮询（亚马逊一般 10 分钟内完成） |
| `1` | 刊登成功 | 进入刊登后处理流程 |
| `2` | 刊登失败 | 查看 `failure_reason`，修改后重新提交 |

轮询策略建议：

```
首次轮询：提交后 30 秒
后续间隔：30s → 60s → 120s（指数退避）
最大等待：15 分钟
超时标记：人工介入
```

失败重试注意：

- 按 `failure_reason` 修改 attributes 后重新调用 publish
- **已成功刊登的商品不要重复提交**

## 9. 刊登后处理流程

### 9.1 等待领星 Listing 同步

刊登成功后，MSKU 不会立刻出现在领星 Listing 页面。

| 阶段 | 预计时间 |
|------|----------|
| 亚马逊处理刊登 | ~10 分钟 |
| 领星 Listing 自动同步 | 15 分钟 ~ 1 小时 |
| 手动触发同步 | 在领星 UI 点「同步」或「单个同步商品」 |

### 9.2 确认 Listing 已同步

```
POST /erp/sc/data/mws/listing
```

```json
{
  "sid": 12518,
  "is_pair": 2,
  "offset": 0,
  "length": 50
}
```

在返回列表中按 MSKU 搜索，确认目标商品已出现。

轮询策略：

```
间隔：5 分钟
最大等待：2 小时
匹配条件：seller_sku == 目标 MSKU
```

### 9.3 MSKU ↔ 本地 SKU 配对

```
POST /erp/sc/storage/product/link
```

```json
{
  "seller_id": "AUF98RFGA31T9",
  "marketplace_id": "ATVPDKIKX0DER",
  "msku": "MLYR-US-NEW-001",
  "sku": "背板更换零件 OTII-230722-0828",
  "is_sync_pic": 1
}
```

| 字段 | 说明 |
|------|------|
| `msku` | 亚马逊 MSKU |
| `sku` | 领星本地 SKU |
| `is_sync_pic` | 是否同步 Listing 图片（1=是） |

### 9.4 完整刊登后状态机

```
PUBLISH_SUBMITTED     已提交刊登
       ↓
PUBLISH_PROCESSING    轮询中（status=0）
       ↓
PUBLISH_SUCCESS       刊登成功（status=1）
       ↓
LISTING_SYNCING       等待领星 Listing 同步
       ↓
LISTING_SYNCED        Listing 已出现
       ↓
SKU_PAIRED            MSKU 配对完成
       ↓
COMPLETED             全流程完成
```

## 11. 商品资料构建规范

### 11.1 无品牌商品

```json
"brand": [
  {
    "value": "Generic",
    "language_tag": "en_US",
    "marketplace_id": "ATVPDKIKX0DER"
  }
]
```

> 部分类目可能仍报错，需参考亚马逊无品牌政策。

### 11.2 无商品编码（UPC 豁免）

前提：已在亚马逊后台申请对应类目的 GTIN 豁免。

```json
"supplier_declared_has_product_identifier_exemption": [
  {
    "value": true,
    "marketplace_id": "ATVPDKIKX0DER"
  }
]
```

注意：

- **不填** `externally_assigned_product_identifier`
- GTIN 豁免按分类分别申请
- 部分分类不支持豁免

### 11.3 变体刊登

**父商品必填：**

```json
"parentage_level": [{ "value": "parent" }],
"child_parent_sku_relationship": [{
  "child_relationship_type": "variation"
}],
"variation_theme": [{ "name": "COLOR" }]
```

**子商品必填：**

```json
"parentage_level": [{ "value": "child" }],
"child_parent_sku_relationship": [{
  "child_relationship_type": "variation",
  "parent_sku": "PARENT-SKU-001"
}],
"variation_theme": [{ "name": "COLOR" }],
"color": [{ "value": "Red", "language_tag": "en_US", "marketplace_id": "ATVPDKIKX0DER" }]
```

注意：

- `variation_theme` 选 COLOR 时，`color` 变为必填
- 选 SIZE 时，`shirt_size` 等变为必填
- 参考 Schema 的 `allOf` 条件和 `$lifecycle.enumDeprecated`（避开已弃用枚举值）

### 11.4 FBM 配送

FBM 新品必须填写：

```json
"merchant_shipping_group": [
  {
    "value": "运费模板ID（非名称）",
    "marketplace_id": "ATVPDKIKX0DER"
  }
]
```

模板 ID 通过 `getMerchantShippingGroup` 接口获取。

## 15. 异常处理与重试策略

### 15.1 领星 API 错误码

| 错误码 | 含义 | 处理 |
|--------|------|------|
| `2001003` | Token 过期 | 重新获取 Token 并重试 |
| `2001006` | 签名错误 | 检查签名算法和参数 |
| `2001007` | 签名过期 | 检查 timestamp 是否为当前时间 |
| `3001002` | IP 不在白名单 | 检查服务器出口 IP |
| `3001008` | 请求超限 | 退避重试 |
| `400` | 参数错误 | 检查参数名和格式 |
| 刊登 status=2 | 亚马逊拒绝 | 解析 failure_reason，人工修正 |

### 15.2 重试策略

| 场景 | 策略 |
|------|------|
| Token 过期 | 立即刷新 Token，重试 1 次 |
| 网络超时 | 最多重试 3 次，间隔 5s/10s/30s |
| 刊登轮询 status=0 | 指数退避，最长 15 分钟 |
| Listing 同步等待 | 每 5 分钟查一次，最长 2 小时 |
| 刊登失败 status=2 | 不自动重试，等人工修正后手动重新发布 |

## 18. 安全规范

### 18.1 凭证管理

| 规则 | 说明 |
|------|------|
| 禁止硬编码 | AppID/AppSecret 不得出现在代码中 |
| 环境变量 | 通过 `.env` 或服务器环境变量注入 |
| 禁止提交 Git | `.env` 加入 `.gitignore` |
| 定期轮换 | 建议每 90 天更换 AppSecret |
| 聊天/文档中不记录明文 Secret | 文档中仅用占位符 |

### 18.2 服务器安全

- 修改默认 root 密码为强密码
- 仅开放必要端口（80/443/22）
- SSH 建议改为密钥登录
- 定期更新系统补丁

### 18.3 API 调用安全

- 所有领星 API 调用必须从白名单 IP 发出
- 生产环境禁止在本地开发机直接调用（本地 IP 不在白名单）
- 记录所有 API 请求/响应日志，便于审计
